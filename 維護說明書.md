## 一、先搞懂三件事

| 名詞 | 白話解釋 |
| --- | --- |
| **GitHub** | 網站「原始檔案」存放的雲端資料夾，網址是 `github.com/jeremy0819/jieceng-web`。所有修改都從這裡開始。 |
| **main 分支** | 正式版本。**只要把修改存進 `main`，機器人就會自動蓋網站。** |
| **gh-pages 分支** | 機器人蓋好的成品放在這裡。GitHub Pages 直接從這裡讀取並對外提供網站。你**不需要**手動碰這個分支。 |
| **GitHub Actions** | 自動化機器人。它會在你存檔後，自動把原始碼蓋成網站、再推送到 `gh-pages` 分支，你什麼都不用做。 |

> ⚠️ **重要：倉庫必須維持「公開（Public）」狀態。**
> 若把倉庫改為私人（Private），GitHub Pages 會立即停止服務、網站變成 404。
> 詳細說明見[第二節技術架構](#為什麼這樣設計)與[第八節 Q5](#q5倉庫改過私人-private現在網站又壞了怎麼辦)。

## 二、網站怎麼自動上線的（含技術架構說明）

### 標準流程圖

```
你在 GitHub 上修改文字／圖片（在 main 分支）
        ↓
按下「Commit（儲存）」到 main 分支
        ↓
機器人 (GitHub Actions) 自動觸發，開始工作（約 2～5 分鐘）：
  步驟 1：下載最新的原始碼
  步驟 2：安裝程式套件（npm ci）
  步驟 3：計算網站的子路徑（/jieceng-web/）
  步驟 4：把整個網站「蓋好」成靜態 HTML 檔案（npm run generate）
  步驟 5：把蓋好的成品推送到 gh-pages 分支
        ↓
GitHub Pages 偵測到 gh-pages 分支有更新，對外更新網站
        ↓
網站自動更新，完成！
```

你**不需要**手動「發佈」或「部署」。**存檔 = 觸發機器人 = 上線**。

> 想看機器人有沒有做完？
> 到 GitHub 倉庫上方的 **Actions** 分頁，看到綠色勾勾 ✅ 就代表成功；紅色叉叉 ❌ 代表有問題，請看[第八節](#八常見問題與排除)。

---

### 為什麼這樣設計？

本網站採用的是「**分支部署法**（Branch Deployment）」，機器人把蓋好的網站推送到 `gh-pages` 分支，GitHub Pages 再從那個分支對外服務。

**這樣設計的原因**：

GitHub 原本還有另一種部署方式叫「**API 部署法**」，它依賴倉庫裡一個叫 `github-pages` 的「環境憑證」才能運作。但這個憑證有一個致命弱點：

> **只要把倉庫改成「私人（Private）」再改回「公開（Public）」，這個憑證就會被 GitHub 自動清空，之後每次部署都會假裝成功、實際上什麼都沒有做，網站持續呈現 404。**

所以我們改用「分支部署法」。這個方法完全不依賴任何環境憑證，就算你不小心切換了倉庫的公私狀態，只要重新設定一次 GitHub Pages 的來源（見下方），一切就會恢復正常。

---

### GitHub Pages 來源設定（首次啟用或恢復時）

> 正常情況下你不需要動這裡。只有在網站無緣無故壞掉時，才來這裡確認一次。

**標準作業程序：**

1. 打開 `https://github.com/jeremy0819/jieceng-web/settings/pages`
2. 找到 **Source（來源）** 區塊
3. 確認下拉選單顯示的是 **`Deploy from a branch`**
   - 如果顯示的是 `GitHub Actions`，請把它改成 `Deploy from a branch`
4. Branch 欄位選 **`gh-pages`**，目錄選 **`/ (root)`**
5. 按 **Save**
6. 等待 1～2 分鐘，重新整理網站

## 三、日常維護：修改文字

### 適用情境

修改作品介紹、首頁標語、關於我們、聯絡頁面上的任何文字內容。

### 重要原則

> ✏️ **只改引號中間的文字，不要動引號、逗號、括號、冒號等符號。**
> 動到符號 = 程式碼語法錯誤 = 網站壞掉。

---

### 修改作品介紹文字（標準作業程序）

**目標檔案**：`composables/useProjects.ts`

**步驟：**

1. 打開 `https://github.com/jeremy0819/jieceng-web`
2. 點進 **`composables`** 資料夾
3. 點擊 **`useProjects.ts`** 檔案名稱，進入檔案瀏覽頁
4. 點右上角的 **鉛筆圖示 ✏️**（Edit this file）進入編輯模式
5. 用瀏覽器的「搜尋」功能（`Ctrl + F` / `Cmd + F`）找到你要改的作品名稱
6. 找到對應欄位，**只改單引號 `'` 中間的文字**，例如：

   ```
   改前：title: '開放建築',
   改後：title: '開放式住宅',
   
   改前：description: '利用自然採光與極簡線條。',
   改後：description: '以自然光線為主角的生活空間。',
   ```

7. 改完後，**捲到頁面最下方**，找到 **Commit changes** 區塊：
   - 第一格（Summary）：填一句說明，例如：`修改開放建築的介紹文字`
   - 第二格（Extended description）：可以留空
   - 確認選擇 **`Commit directly to the main branch`** ← 這個選項很重要
   - 按綠色按鈕 **`Commit changes`**

8. **確認點**：存檔成功後，頁面會跳回檔案瀏覽頁。到上方 **Actions** 分頁確認最新一筆出現綠色勾勾 ✅。

9. 等 2～5 分鐘，重新整理網站即可看到更新。

---

操作方法完全相同：找到檔案 → 點鉛筆 → 只改文字 → Commit to main。

## 四、日常維護：換圖片

所有圖片都放在 **`public/image/`** 資料夾。

### 換掉現有圖片（標準作業程序）

**原則：新圖片檔名要與舊圖片完全一致，上傳後自動覆蓋。**

**步驟：**

1. **先確認舊圖片的檔名**：進入 `public/image/` 資料夾，記下要替換的圖片檔名（例如 `project-1.jpg`）

2. **準備新圖片**：
   - 在你的電腦上把新圖片重新命名，讓它和舊圖片的檔名**完全一樣**（例如 `project-1.jpg`）
   - 檔名規則：全英文小寫、不含中文、不含空格（`project-5.jpg` ✅、`作品五.jpg` ❌、`Project 5.JPG` ❌）
   - 建議規格：寬度 1600px 以上，檔案大小壓縮到 500KB 以內

3. **上傳圖片**：
   - 在 `public/image/` 資料夾頁面，點右上角 **`Add file` → `Upload files`**
   - 把新圖片拖進上傳區域，或點 `choose your files` 選擇檔案
   - 等待上傳完成（進度條消失）

4. **儲存**：
   - 下方 **Commit changes** 區塊填說明，例如：`更換 project-1 圖片`
   - 選 **`Commit directly to the main branch`**
   - 按 **`Commit changes`**

5. **確認點**：到 **Actions** 分頁等待綠色勾勾 ✅，再等 2～5 分鐘重新整理網站確認圖片更新。

---

### 新增全新圖片

步驟與上方相同，差別是：
- 新圖片不需要與任何舊檔名相符，自行命名即可（英文小寫）
- 新增完圖片後，還需要在 `useProjects.ts` 裡把 `image` 欄位的路徑改成新的檔名（見[第五節](#五日常維護新增一個作品)）

## 五、日常維護：新增一個作品

新增一個作品需要**同時修改三個地方**，缺一不可。建議直接請 Claude 幫你做（見[第七節](#七用-claude-來幫你維護含-skill-管理)），它已內建完整的「新增作品」自動流程。若要手動操作，請嚴格依照以下步驟：

---

### 步驟 1：上傳作品圖片

依照[第四節新增圖片流程](#新增全新圖片)，把圖片上傳到 `public/image/`。

**命名建議**：`project-5.jpg`（數字對應新作品的 id，見步驟 2）

**確認點**：在 `public/image/` 資料夾能看到新的圖片檔案。

---

### 步驟 2：新增作品資料

**目標檔案**：`composables/useProjects.ts`

1. 打開 `useProjects.ts`，點鉛筆 ✏️ 進入編輯模式
2. 捲到檔案**最底部**，找到最後一個 `},` 的後面
3. 在最後一個 `},` 的**下方**新增一段（注意縮排與逗號位置）：

```ts
  {
    id: 5,
    title: '新作品名稱',
    category: 'Residential',
    description: '一句話短介紹（顯示在作品卡片上，建議 20 字以內）。',
    fullDescription: '詳細完整介紹（顯示在作品詳情頁，可以多行）。',
    image: 'image/project-5.jpg',
    specs: {
      Location: '台北市, 信義區',
      Year: '2026',
      Area: '120 sqm',
      System: '工法名稱',
    },
  },
```

**欄位說明：**

| 欄位 | 說明 | 注意事項 |
| --- | --- | --- |
| `id` | 作品編號 | 目前最大是 4，新的填 **5**，下一個填 6，依此類推 |
| `title` | 作品名稱 | 顯示在卡片和詳情頁標題 |
| `category` | 分類 | **只能**填這五個之一：`Residential`（住宅）、`Commercial`（商業）、`Landscape`（景觀）、`Renovation`（改造） |
| `description` | 卡片短描述 | 約 20 字以內 |
| `fullDescription` | 詳情頁完整描述 | 可以寫多行，注意每行用 `\n` 換行或直接在引號內換行 |
| `image` | 圖片路徑 | 開頭**不加斜線**，格式為 `image/檔名.jpg` |
| `specs` | 規格資訊 | Location、Year、Area、System 四項都要填 |

4. 改完後，Commit to main，填說明例如：`新增作品：山居歲月`

**確認點**：存檔後確認檔案內容格式正確，所有引號、逗號都成對。

---

### 步驟 3：登記新頁面路由

**目標檔案**：`nuxt.config.ts`

> ⚠️ **這步驟非常容易被忘記。漏掉這步，新作品的詳情頁部署後會出現 404。**

1. 打開 `nuxt.config.ts`，點鉛筆 ✏️ 進入編輯模式
2. 用搜尋（`Ctrl + F`）找到 `'/project/4'` 這行
3. 在它的**下方**新增一行：`'/project/5',`（數字對應你的新 id）

   ```
   改前：
       '/project/4',
     ]
   
   改後：
       '/project/4',
       '/project/5',
     ]
   ```

4. Commit to main，填說明例如：`登記 project 5 路由`

**確認點**：確認新增的那行有**逗號結尾**，且沒有多餘的符號。

---

### 步驟 4：最終確認

三個檔案全部 Commit 到 `main` 後：

1. 到 **Actions** 分頁等待最新一筆出現綠色勾勾 ✅（約 2～5 分鐘）
2. 打開 `https://jeremy0819.github.io/jieceng-web/portfolio` 確認新作品出現
3. 點進新作品的詳情頁，確認不是 404

### 什麼是 Skill（技能）？

**Skill 就是預先寫好的「標準作業程序說明書」**，存成檔案放在專案裡。Claude 看到相關需求時，會自動照著這份說明書執行，確保每次都按正確流程、不會漏步驟。

本專案已經內建了一個 Skill：**`add-project`（新增作品）**。當你說「新增作品」時，Claude 會自動載入它，照著「確認資訊 → 上傳圖片 → 改作品資料 → 登記路由 → 驗證」的標準流程幫你做完。

### Skill 存在哪裡？

```
.claude/
└── skills/
    └── add-project/
        └── SKILL.md   ← 技能的內容（步驟說明）寫在這裡
```

每一個 Skill 就是一個子資料夾，裡面放一個 `SKILL.md` 檔案。

---

### Skill 管理標準作業程序

#### ① 查看現有的 Skill

進入 GitHub 的 `.claude/skills/` 資料夾，每個子資料夾就是一個技能，點進去看 `SKILL.md` 就是內容。

#### ② 修改現有 Skill

1. 打開 `.claude/skills/<技能名稱>/SKILL.md`
2. 點鉛筆 ✏️ 進入編輯模式
3. 修改步驟說明文字
4. Commit to main

#### ③ 新增一個 Skill（標準作業程序）

1. 進入 `.claude/skills/` 資料夾
2. 點右上角 **`Add file` → `Create new file`**
3. 在「Name your file...」欄位輸入：`你的技能名稱/SKILL.md`
   - 注意：打入斜線 `/` 時，GitHub 會自動在前面建立資料夾
4. 在檔案內容的**最上方**，貼入以下標頭（Claude 必須看到這個才能識別這個 Skill）：

   ```
   ---
   name: 技能英文代號（例：update-contact）
   description: 一句話說明這個技能在做什麼，以及使用者會用什麼話來要求（例：修改聯絡資訊，包含電話、地址、Email、社群連結時使用）。
   ---
   ```

5. 標頭下方用中文寫出「要照什麼步驟做」，可以參考現有 `add-project/SKILL.md` 的格式
6. Commit to main

> 💡 `description` 寫得越清楚（包含「使用者會怎麼開口要求」），Claude 越能在對的時機自動使用這個技能。

#### ④ 刪除 Skill

1. 打開該技能的 `SKILL.md`
2. 點右上角的 **`...`（三個點）** → **`Delete file`**
3. Commit to main

## 八、常見問題與排除

---

**Q1：網站打開是 404 / 空白頁？**

按照以下清單逐一確認：

- [ ] 網址是否正確：`https://jeremy0819.github.io/jieceng-web/`（結尾的 `/jieceng-web/` 不能少）
- [ ] 到 **Actions** 分頁確認最新一筆是否為綠色 ✅（若是紅色 ❌，見 Q3）
- [ ] 到 `Settings → Pages` 確認 Source 是 `Deploy from a branch`，Branch 是 `gh-pages`，目錄是 `/ (root)`
- [ ] 若上述都對，等 5 分鐘後強制重新整理（`Ctrl + F5` / `Cmd + Shift + R`）

---

**Q2：改了文字但網站沒有更新？**

- 機器人需要 2～5 分鐘。先到 **Actions** 分頁確認有綠色勾勾 ✅。
- 瀏覽器可能有快取，按 `Ctrl + F5`（Mac 是 `Cmd + Shift + R`）強制重新整理。
- 如果 Actions 顯示成功但網站確實沒變，確認你是不是 Commit 到 `main` 而不是其他分支。

---

**Q3：Actions 出現紅色叉叉 ❌，不知道原因？**

1. 點進那筆紅色的工作流程記錄
2. 點 **`deploy`** 那個步驟，展開看哪一行出錯（通常是紅色文字）
3. 最常見原因：修改 `useProjects.ts` 或 `nuxt.config.ts` 時，不小心刪掉或多加了符號（逗號、引號、括號）
4. 修正方法：把出問題的檔案內容貼給 Claude，請它找出哪裡符號錯了並修正

**Q4：我不小心改壞了，想還原到改壞之前的版本？**

**Q5：倉庫改過「私人（Private）」，現在網站又壞了怎麼辦？**

這是**最常見的部署失效原因**，原理說明如下：

> 當你把倉庫設定改為 Private，GitHub 會自動停用 GitHub Pages 並清空所有部署記錄。即使你改回 Public，GitHub Pages 的來源設定也可能已被重置，導致網站持續顯示 404。

**標準還原程序（依序執行）：**

1. **確認倉庫是 Public**：
   - 打開 `Settings`（倉庫設定頁）
   - 捲到最底部的 **Danger Zone**
   - 確認顯示的是「This repository is currently public」
   - 如果不是，點 **`Change repository visibility`** 改回 Public

2. **重新設定 GitHub Pages 來源**：
   - 打開 `Settings → Pages`
   - **Source** 選 **`Deploy from a branch`**
   - **Branch** 選 **`gh-pages`**，目錄選 **`/ (root)`**
   - 按 **Save**

3. **確認 `gh-pages` 分支存在**：
   - 在倉庫主頁，點分支下拉選單（通常顯示「main」），看是否有 `gh-pages` 分支
   - 若有：Pages 設定好後等 1～2 分鐘，網站應恢復
   - 若沒有：需要觸發一次部署。到 **Actions** 分頁，點最新一筆工作流程右側的 **`Re-run all jobs`**（重新執行），等待完成

4. **確認點**：打開網站網址，確認能正常顯示首頁。

## 十、部署架構技術摘要（給未來維護工程師）

本網站使用 **Nuxt 3 SSG（Static Site Generation）** 架構，部署於 GitHub Pages。

**部署流程：**

1. 開發者 push 到 `main` 分支
2. GitHub Actions 觸發 `.github/workflows/deploy.yml`
3. 執行 `npm run generate`，產生靜態檔案至 `.output/public/`
4. 使用 `peaceiris/actions-gh-pages@v4` 將靜態檔案推送到 `gh-pages` 分支
5. GitHub Pages 從 `gh-pages` 分支提供服務

**為何使用 peaceiris 而非 actions/deploy-pages：**

GitHub 原生的 `actions/upload-pages-artifact` + `actions/deploy-pages` 方案依賴 `github-pages` **Environment**（倉庫環境憑證）。此憑證在倉庫 visibility 從 Public → Private → Public 切換時會被清空，導致後續部署靜默失敗（workflow 顯示成功但 Pages 未更新）。

`peaceiris/actions-gh-pages` 透過 git push 方式部署，不依賴 Environment，對倉庫設定變更有更強的容錯性。

**子路徑設定：**

由於倉庫名稱為 `jieceng-web`（非 `<username>.github.io`），GitHub Pages 自動將網站掛載在 `/jieceng-web/` 子路徑。部署腳本透過 `DEPLOY_BASE` 環境變數將此路徑傳入 Nuxt，確保所有資源路徑正確。
