## 一、先搞懂三件事

| 名詞 | 白話解釋 |
| --- | --- |
| **GitHub** | 網站「原始檔案」存放的雲端資料夾，網址是 `github.com/jasonhouse2007-glitch/jieceng-web`。所有修改都從這裡開始。 |
| **main 分支** | 正式版本。**只要把修改存進 `main`，機器人就會自動蓋網站。** |
| **gh-pages 分支** | 機器人蓋好的成品放在這裡。GitHub Pages 直接從這裡讀取並對外提供網站。你**不需要**手動碰這個分支。 |
| **GitHub Actions** | 自動化機器人。它會在你存檔後，自動把原始碼蓋成網站、再推送到 `gh-pages` 分支，你什麼都不用做。 |

> ⚠️ **重要：倉庫必須維持「公開（Public）」狀態。**
> 若把倉庫改為私人（Private），GitHub Pages 會立即停止服務、網站變成 404。
> 詳細說明見[第二節技術架構](#為什麼這樣設計)與[第八節 Q5](#q5倉庫改過私人-private現在網站又壞了怎麼辦)。

## 二、網站怎麼自動上線的（含技術架構說明）

### 標準流程圖

```
你在 GitHub 上修改文字／圖片（在 main 分支）
        ↓
按下「Commit（儲存）」到 main 分支
        ↓
機器人 (GitHub Actions) 自動觸發，開始工作（約 2～5 分鐘）：
  步驟 1：下載最新的原始碼
  步驟 2：安裝程式套件（npm ci）
  步驟 3：計算網站的子路徑（/jieceng-web/）
  步驟 4：把整個網站「蓋好」成靜態 HTML 檔案（npm run generate）
  步驟 5：把蓋好的成品推送到 gh-pages 分支
        ↓
GitHub Pages 偵測到 gh-pages 分支有更新，對外更新網站
        ↓
網站自動更新，完成！
```

你**不需要**手動「發佈」或「部署」。**存檔 = 觸發機器人 = 上線**。

> 想看機器人有沒有做完？
> 到 GitHub 倉庫上方的 **Actions** 分頁，看到綠色勾勾 ✅ 就代表成功；紅色叉叉 ❌ 代表有問題，請看[第八節](#八常見問題與排除)。

---

### 為什麼這樣設計？

本網站採用的是「**分支部署法**（Branch Deployment）」，機器人把蓋好的網站推送到 `gh-pages` 分支，GitHub Pages 再從那個分支對外服務。

**這樣設計的原因**：

GitHub 原本還有另一種部署方式叫「**API 部署法**」，它依賴倉庫裡一個叫 `github-pages` 的「環境憑證」才能運作。但這個憑證有一個致命弱點：

> **只要把倉庫改成「私人（Private）」再改回「公開（Public）」，這個憑證就會被 GitHub 自動清空，之後每次部署都會假裝成功、實際上什麼都沒有做，網站持續呈現 404。**

所以我們改用「分支部署法」。這個方法完全不依賴任何環境憑證，就算你不小心切換了倉庫的公私狀態，只要重新設定一次 GitHub Pages 的來源（見下方），一切就會恢復正常。

---

### GitHub Pages 來源設定（首次啟用或恢復時）

> 正常情況下你不需要動這裡。只有在網站無緣無故壞掉時，才來這裡確認一次。

**標準作業程序：**

1. 打開 `https://github.com/jeremy0819/jieceng-web/settings/pages`
2. 找到 **Source（來源）** 區塊
3. 確認下拉選單顯示的是 **`Deploy from a branch`**
   - 如果顯示的是 `GitHub Actions`，請把它改成 `Deploy from a branch`
4. Branch 欄位選 **`gh-pages`**，目錄選 **`/ (root)`**
5. 按 **Save**
6. 等待 1～2 分鐘，重新整理網站

## 三、日常維護：修改文字

### 適用情境

修改作品介紹、首頁標語、關於我們、聯絡頁面上的任何文字內容。

### 重要原則

> ✏️ **只改引號中間的文字，不要動引號、逗號、括號、冒號等符號。**
> 動到符號 = 程式碼語法錯誤 = 網站壞掉。

---

### 修改作品介紹文字（標準作業程序）

**目標檔案**：`composables/useProjects.ts`

**步驟：**

1. 打開 `https://github.com/jeremy0819/jieceng-web`
2. 點進 **`composables`** 資料夾
3. 點擊 **`useProjects.ts`** 檔案名稱，進入檔案瀏覽頁
4. 點右上角的 **鉛筆圖示 ✏️**（Edit this file）進入編輯模式
5. 用瀏覽器的「搜尋」功能（`Ctrl + F` / `Cmd + F`）找到你要改的作品名稱
6. 找到對應欄位，**只改單引號 `'` 中間的文字**，例如：

   ```
   改前：title: '開放建築',
   改後：title: '開放式住宅',
   
   改前：description: '利用自然採光與極簡線條。',
   改後：description: '以自然光線為主角的生活空間。',
   ```

7. 改完後，**捲到頁面最下方**，找到 **Commit changes** 區塊：
   - 第一格（Summary）：填一句說明，例如：`修改開放建築的介紹文字`
   - 第二格（Extended description）：可以留空
   - 確認選擇 **`Commit directly to the main branch`** ← 這個選項很重要
   - 按綠色按鈕 **`Commit changes`**

8. **確認點**：存檔成功後，頁面會跳回檔案瀏覽頁。到上方 **Actions** 分頁確認最新一筆出現綠色勾勾 ✅。

9. 等 2～5 分鐘，重新整理網站即可看到更新。

## 四、日常維護：換圖片

所有圖片都放在 **`public/image/`** 資料夾。

### 換掉現有圖片（標準作業程序）

**原則：新圖片檔名要與舊圖片完全一致，上傳後自動覆蓋。**

**步驟：**

1. **先確認舊圖片的檔名**：進入 `public/image/` 資料夾，記下要替換的圖片檔名（例如 `project-1.jpg`）

2. **準備新圖片**：
   - 在你的電腦上把新圖片重新命名，讓它和舊圖片的檔名**完全一樣**（例如 `project-1.jpg`）
   - 檔名規則：全英文小寫、不含中文、不含空格（`project-5.jpg` ✅、`作品五.jpg` ❌、`Project 5.JPG` ❌）
   - 建議規格：寬度 1600px 以上，檔案大小壓縮到 500KB 以內

3. **上傳圖片**：
   - 在 `public/image/` 資料夾頁面，點右上角 **`Add file` → `Upload files`**
   - 把新圖片拖進上傳區域，或點 `choose your files` 選擇檔案
   - 等待上傳完成（進度條消失）

4. **儲存**：
   - 下方 **Commit changes** 區塊填說明，例如：`更換 project-1 圖片`
   - 選 **`Commit directly to the main branch`**
   - 按 **`Commit changes`**

5. **確認點**：到 **Actions** 分頁等待綠色勾勾 ✅，再等 2～5 分鐘重新整理網站確認圖片更新。

---

### 新增全新圖片

步驟與上方相同，差別是：
- 新圖片不需要與任何舊檔名相符，自行命名即可（英文小寫）
- 新增完圖片後，還需要在 `useProjects.ts` 裡把 `image` 欄位的路徑改成新的檔名（見[第五節](#五日常維護新增一個作品)）

### 步驟 1：上傳作品圖片

依照[第四節新增圖片流程](#新增全新圖片)，把圖片上傳到 `public/image/`。

**命名建議**：`project-5.jpg`（數字對應新作品的 id，見步驟 2）

**確認點**：在 `public/image/` 資料夾能看到新的圖片檔案。

### 步驟 2：新增作品資料

**目標檔案**：`composables/useProjects.ts`

1. 打開 `useProjects.ts`，點鉛筆 ✏️ 進入編輯模式
2. 捲到檔案**最底部**，找到最後一個 `},` 的後面
3. 在最後一個 `},` 的**下方**新增一段（注意縮排與逗號位置）：

```ts
  {
    id: 3,
    title: '傑丞。新建案名稱',
    year: 2026,
    category: 'residential',
    categoryLabel: '住宅建築',
    location: '桃園市, 桃園區',
    description: '一句話短介紹（顯示在作品卡片上，建議 20 字以內）。',
    fullDescription: '詳細完整介紹（顯示在建案詳情頁，可以多行）。',
    image: 'image/project-3.jpg',
    highlights: [
      '亮點一（例：採用百年工法，結構耐震達 7 級）',
      '亮點二（例：全棟導入 iHome 智慧系統）',
      '亮點三（例：節能外牆，降低 40% 冷暖氣耗能）',
      '亮點四（例：建物履歷全程可溯源，交屋即附電子檔）',
    ],
    specs: {
      '地點': '桃園市桃園區',
      '年份': '2026',
      '樓地板面積': '320 坪',
      '工法': 'iHome 5.0 全系統',
    },
    technologies: ['centennial', 'healthy', 'energy'],
  },
```

**欄位說明：**

| 欄位 | 說明 | 注意事項 |
| --- | --- | --- |
| `id` | 建案編號（數字） | 目前最大是 **2**，新的填 **3**，下一個填 4，依此類推 |
| `title` | 建案名稱 | 慣例格式「傑丞。建案名稱」 |
| `year` | 完工年份（數字） | **不加引號**，例如 `2026`（不是 `'2026'`） |
| `category` | 分類代碼 | 填 `'residential'`（住宅）或 `'commercial'`（商業） |
| `categoryLabel` | 分類中文名稱 | 顯示在卡片和詳情頁標籤上 |
| `location` | 地點 | 格式「城市, 行政區」，例如 `'桃園市, 桃園區'` |
| `description` | 卡片短描述 | 約 20 字以內 |
| `fullDescription` | 詳情頁完整描述 | 可以寫多行 |
| `image` | 圖片路徑 | 開頭**不加斜線**，格式為 `image/檔名.jpg` |
| `highlights` | 四大亮點（陣列） | **必須是 4 項**，每項約 15～25 字 |
| `specs` | 規格資訊（物件） | Key 是欄位名稱，Value 是內容，可自由增減欄位 |
| `technologies` | iHome 五大宅工法 | 填入該建案採用的工法代碼（見下表）；**必須至少一項** |

**`technologies` 可用代碼（iHome 五大宅）：**

| 代碼 | 中文名稱 | 說明 |
| --- | --- | --- |
| `'centennial'` | 百年宅 | 百年結構工法，耐震耐久 |
| `'healthy'` | 健康宅 | 健康建材，抑菌防霉 |
| `'energy'` | 節能宅 | 節能外牆與設備 |
| `'smart'` | 智慧宅 | 智慧家居系統 |
| `'resume'` | 履歷宅 | 建物履歷全程可溯源 |

> 💡 **`technologies` 欄位的作用**：決定這個建案會出現在「全案管理案例」頁面的哪些篩選標籤下。填越多工法，這個建案在越多篩選條件下都能被看到。

4. 改完後，Commit to main，填說明例如：`新增建案：傑丞印象天河`

**確認點**：存檔後確認檔案內容格式正確，所有引號、逗號都成對。

### 步驟 3：登記新頁面路由

**目標檔案**：`nuxt.config.ts`

> ⚠️ **這步驟非常容易被忘記。漏掉這步，新作品的詳情頁部署後會出現 404。**

1. 打開 `nuxt.config.ts`，點鉛筆 ✏️ 進入編輯模式

3. 在它的**下方**新增一行：`'/project/3',`（數字對應你的新 id）

   ```

       '/project/2'
     ]
   
   改後：
       '/project/2',
       '/project/3',
     ]
   ```

4. Commit to main，填說明例如：`登記 project 3 路由`

**確認點**：確認新增的那行有**逗號結尾**，且沒有多餘的符號。

## 七、用 Claude 來幫你維護（含 Skill 管理）

### 什麼是 Skill（技能）？

**Skill 就是預先寫好的「標準作業程序說明書」**，存成檔案放在專案裡。Claude 看到相關需求時，會自動照著這份說明書執行，確保每次都按正確流程、不會漏步驟。

本專案已經內建了一個 Skill：**`add-project`（新增作品）**。當你說「新增作品」時，Claude 會自動載入它，照著「確認資訊 → 上傳圖片 → 改作品資料 → 登記路由 → 驗證」的標準流程幫你做完。

### Skill 存在哪裡？

```
.claude/
└── skills/
    └── add-project/
        └── SKILL.md   ← 技能的內容（步驟說明）寫在這裡
```

## 十、部署架構技術摘要（給未來維護工程師）

本網站使用 **Nuxt 3 SSG（Static Site Generation）** 架構，部署於 GitHub Pages。

**部署流程：**

1. 開發者 push 到 `main` 分支
2. GitHub Actions 觸發 `.github/workflows/deploy.yml`
3. 執行 `npm run generate`，產生靜態檔案至 `.output/public/`
4. 使用 `peaceiris/actions-gh-pages@v4` 將靜態檔案推送到 `gh-pages` 分支
5. GitHub Pages 從 `gh-pages` 分支提供服務

**為何使用 peaceiris 而非 actions/deploy-pages：**

GitHub 原生的 `actions/upload-pages-artifact` + `actions/deploy-pages` 方案依賴 `github-pages` **Environment**（倉庫環境憑證）。此憑證在倉庫 visibility 從 Public → Private → Public 切換時會被清空，導致後續部署靜默失敗（workflow 顯示成功但 Pages 未更新）。

`peaceiris/actions-gh-pages` 透過 git push 方式部署，不依賴 Environment，對倉庫設定變更有更強的容錯性。

**子路徑設定：**

由於倉庫名稱為 `jieceng-web`（非 `<username>.github.io`），GitHub Pages 自動將網站掛載在 `/jieceng-web/` 子路徑。部署腳本透過 `DEPLOY_BASE` 環境變數將此路徑傳入 Nuxt，確保所有資源路徑正確。

## 十一、GitHub 倉庫所有權轉移 SOP

當需要把倉庫從一個 GitHub 帳號轉移到另一個帳號時，務必使用「正式轉移所有權」，**不要用 Fork**。

> **為什麼不用 Fork？**
> Fork 會與原帳號建立上游關聯，之後任何 PR 都會預設指回原帳號；且 Fork 出來的 repo 名稱若不同（如 `-`），子路徑就會跑掉，導致網站整個壞掉。

### 步驟 0：先刪掉公司帳號的舊 Fork（如有）
1. 進入公司帳號下的舊 Fork repo
2. Settings → 最下方 Danger Zone → **Delete this repository**
3. 輸入 repo 名稱確認刪除

✅ **確認點**：公司帳號下沒有名為 `jieceng-web` 的 repo

### 步驟 1：從原帳號發起轉移
1. 用原帳號登入，進入 `github.com/原帳號/jieceng-web`
2. **Settings → General → 最下方 Danger Zone → Transfer ownership**
3. 點 **Transfer**，在「New owner」欄填入公司帳號名稱
4. 輸入 repo 名稱 `jieceng-web` 確認 → 送出

### 步驟 2：公司帳號接受轉移
1. 用公司帳號登入，收到 GitHub 通知
2. 點 **Accept transfer**

✅ **確認點**：網址變成 `github.com/公司帳號/jieceng-web`

### 步驟 3：重新開啟 GitHub Pages
1. 進入新帳號的 `jieceng-web` → **Settings → Pages**
2. Source：**Deploy from a branch**
3. Branch：**`gh-pages`** / **`/ (root)`** → Save

✅ **確認點**：因為 repo 名稱未變（仍是 `jieceng-web`），`gh-pages` 分支的所有路徑都正確，設好後網站立即上線。

### 步驟 4：確認 Actions 仍運作
1. 進 **Actions** 分頁，若看到提示就點 **Enable workflows**
2. 之後 push 到 `main` 即可觸發自動部署

### 重要提醒
- **舊網址失效**：GitHub 會自動轉址舊 URL 到新位置，但建議更新所有文件中的 URL
- **不需要額外 Secrets**：本專案只用內建 `GITHUB_TOKEN`
- **Claude Code 環境需重建**：轉移後請用公司帳號在 Claude Code Web 建立新環境，指向新 repo，才能繼續使用 AI 維護

---

> **維護時的黃金守則：**
> 1. **一次只改一個地方**，改完確認 Actions ✅ 後，再改下一個。
> 2. **只改文字內容**，不要動任何符號（引號、逗號、括號）。
> 3. **倉庫永遠保持 Public**，不要改成 Private。
> 4. 不確定的時候，把檔案內容貼給 Claude 問它，比自己硬改安全得多。

## 十五、AI 功能說明（暫緩整合）

舊版 React 原始碼中包含兩個 AI 功能：**AI 建築諮詢（Gemini Chat）** 和 **AI 圖像生成（Image Editor）**，目前**刻意未整合**進現行網站，原因如下：

### 為什麼不整合？

本網站採用 **SSG 靜態網站**架構，所有程式碼在部署時會被編譯成純 HTML/CSS/JS 檔案，**API 金鑰如果放在前端程式碼裡，任何人都能在瀏覽器開發者工具中看到**，造成嚴重的安全與費用風險（他人可盜用你的 API 金鑰無限呼叫，費用由你承擔）。

### 若未來想要 AI 功能，怎麼辦？

需要建立一個**後端 API 代理**，讓 Gemini API 金鑰只存在後端，前端只與後端溝通。可選方案：

| 方案 | 費用 | 難度 |
| --- | --- | --- |
| Cloudflare Workers | 免費額度大 | 需工程師設定 |
| Vercel Functions | 免費額度大 | 需工程師設定 |
| 獨立後端伺服器 | 需租主機 | 需工程師維護 |

> 💡 若公司決定整合 AI 功能，請交由工程師處理，**切勿**直接把 `GEMINI_API_KEY` 貼進前端程式碼。舊版 React 原始碼保存在 `ai-studio-old` 分支供工程師參考介面設計。

## 十六、網站設計規範（工程師維護參考）

本節整理本網站採用的核心設計原則，供日後維護或擴充頁面的工程師遵循，確保新增元素與現有風格一致。

---

### 邊框：用 ring 取代 border（避免 shadow 髒掉）

**原則**：任何有 `shadow` 的元素，邊框一律改用 `ring`，不用 `border`。

`border` 是「嵌在」元素邊界上的實色線，當元素同時有 `shadow` 時，陰影與邊框之間會產生視覺上的「混濁帶」。改用 `ring`（本質是 `box-shadow`）放在元素外側，陰影過渡更銳利。

**寫法對照：**

```html
<!-- ❌ 舊寫法：shadow + border，視覺混濁 -->
<div class="shadow-xl border border-stone-100 rounded-2xl">

<!-- ✅ 新寫法：shadow + outer ring，視覺乾淨 -->
<div class="shadow-xl ring-1 ring-black/[0.07] rounded-2xl">
```

**本站統一規範：**

| 元素 | ring 設定 |
| --- | --- |
| 圖片容器（淺色背景，有 shadow） | `ring-1 ring-black/[0.07]` |
| 圖片容器（淺色背景，無 shadow） | `ring-1 ring-black/[0.06]` |
| 下拉選單面板（有 shadow-xl） | `ring-1 ring-black/[0.07]` |
| CTA 按鈕（深色背景） | `ring-1 ring-warm-white/35` |
| CTA 按鈕（淺色背景） | `ring-1 ring-charcoal/20` |
| 工法標籤（深色背景） | `ring-1 ring-white/10` |

hover 狀態要同步把 ring 顏色改為填色，例如：
```html
hover:bg-charcoal hover:ring-charcoal
```

---

### 按鈕：pill 形、ring 邊框、高度 36-38px

```html
<!-- CTA 按鈕標準寫法 -->
<button class="px-8 py-3.5 ring-1 ring-charcoal/20 rounded-full text-xs tracking-widest uppercase hover:bg-charcoal hover:text-warm-white hover:ring-charcoal transition-all duration-700 elegant-transition">
  行動呼籲
</button>
```

- 高度由 `py-3.5`（14px 上下 padding）控制，約等於 36-38px
- 形狀一律 `rounded-full`（pill 形）
- 不用固定 `height`，由 padding 控制

---

### 標題排版：大字要收緊 tracking

- **Hero 等級（text-7xl 以上）**：`tracking-tight`
- **Section 標題（text-4xl～6xl）**：`tracking-tight` 或 `leading-tight`
- **卡片標題（text-2xl～3xl）**：可用 `tracking-zh`（0.05em）
- **中文 eyebrow（最小字）**：`font-mono uppercase tracking-[0.35em～0.5em]`

字越大，字母間距在視覺上被放大，`tracking-tight` 讓標題更有衝擊力。

---

### Section 標題：Inline Heading 風格

重要 section 不用傳統「大標題 + 換行 + 副標」，改成標題與說明文字並排：

```html
<div class="mb-14 md:mb-20">
  <span class="font-mono text-emerald-brand text-xs tracking-[0.35em] uppercase mb-6 block">
    Section Label
  </span>
  <div class="flex flex-col lg:flex-row lg:items-baseline gap-4 lg:gap-10">
    <h2 class="font-serif text-5xl md:text-6xl font-light tracking-tight text-charcoal shrink-0">
      主標題
    </h2>
    <p class="font-sans text-base md:text-lg font-light text-charcoal/40 max-w-[38ch] leading-relaxed tracking-zh">
      副標說明文字，約 38 個字元寬，自然換行。
    </p>
  </div>
</div>
```

副標文字寬度用 `max-w-[38ch]`（ch = 字元寬度），讓行寬不管字體大小都保持在舒適的閱讀範圍。

---

### Canvas Grid：全版寬橫線分隔 Section

在 section 加上 `vw-sep` class，可以讓頂部分隔線延伸到視窗滿寬（突破 container 邊界），產生 Stripe / Tailwind 官網風格的精緻網格感。

```html
<!-- 用法：直接加在 section class 上 -->
<section class="section-spacing bg-warm-white vw-sep">
  ...
</section>
```

CSS 定義在 `assets/css/main.css`：
```css
.vw-sep::before {
  content: '';
  position: absolute;
  top: 0;
  left: 50%; transform: translateX(-50%);
  width: 100vw;
  height: 1px;
  background: #e7e5e4;
}
```

**注意**：若 section 的父元素有 `overflow: hidden`，`vw-sep` 的線條會被截斷。確認父層沒有 `overflow: hidden` 後再使用。

---

### Hero：左對齊 + 內容置底

AI 產生的 Hero 預設是置中。本站 Hero 使用左對齊 + 內容置底風格：

```html
<section class="relative h-[95vh] flex items-end">
  <!-- 背景圖 -->
  <div class="absolute inset-0">...</div>

  <!-- 內容：底部左對齊 -->
  <div class="relative z-10 w-full pb-20 md:pb-28">
    <div class="container-custom">
      <p class="eyebrow">...</p>
      <h1 class="...tracking-tight...">品牌名稱</h1>
      <div class="flex items-center gap-12">
        <p>Tagline</p>
        <NuxtLink class="...ring-1...rounded-full...">CTA</NuxtLink>
      </div>
    </div>
  </div>

  <!-- Scroll 指示：右下角 -->
  <div class="absolute bottom-10 right-14">...</div>
</section>
```

漸層方向改為 `bg-gradient-to-t`（由下往上），底部加深確保文字可讀性：`from-charcoal/90 via-charcoal/30 to-charcoal/10`。
